# 環境構築・デプロイ運用ガイド（本番運用しながら開発する）

本番（production）を運用しつつ、安全に修正・新機能開発を行うための環境構成と手順。

> **大原則**: demo 環境の環境変数は **すべて demo のリソース（demo Firebase / demo LINE / demo Vercel）を指す値**にする。
> 本番の値は demo に1つも入れない。これさえ守れば、demo での操作が本番に影響することはない。

---

## 1. 環境マトリクス

| 層 | 用途 | Git ブランチ | デプロイ先 | `detectEnv()` 判定 | LINE チャネル | Firebase |
|----|------|-------------|-----------|--------------------|--------------|----------|
| local | 手元開発 | 任意 | `localhost:3000` | dev | demo | demo |
| **demo** | 結合確認・関係者テスト | `develop` | Vercel demo | review（`*.vercel.app`）/ prod（独自ドメイン時） | **demo** | **demo** |
| production | 本番 | `main` | Vercel 本番 | prod | 本番 | 本番 |

- `detectEnv()`（`src/lib/liff.ts`）: `localhost`→dev / `*.vercel.app`→review / それ以外（独自ドメイン）→prod。`?env=dev|review|prod` でも明示可。
- **prod は dev LIFF ID へフォールバックしない**（本番で開発チャネル誤用を防止済み）。
- ブランチ運用: `develop` で開発 → demo で確認 → 問題なければ `main` に fast-forward → 本番反映。

---

## 2. いま用意済みのもの（前提）

- Vercel: demo 環境
- Firebase: demo プロジェクト
- LINE: demo ミニアプリ（LIFF + Login、必要なら Messaging API チャネル）

残作業は **環境変数を demo の値で設定し、各サービスの URL を demo に向け、検証する**こと。

---

## 3. 環境変数の取得元と設定先

`.env.local.example` に全キーと取得先が記載済み。**local は `.env.local`、demo は Vercel demo の Environment Variables** に設定する。
`NEXT_PUBLIC_*` はビルド時に焼き込まれるため、変更後は **再デプロイが必要**。

### 3-1. Firebase（demo プロジェクト）
Firebase Console > プロジェクト設定 > サービスアカウント > 「新しい秘密鍵を生成」で JSON を取得し、以下にマッピング:

| 変数 | 値 |
|------|----|
| `FIREBASE_PROJECT_ID` | JSON の `project_id` |
| `FIREBASE_CLIENT_EMAIL` | JSON の `client_email` |
| `FIREBASE_PRIVATE_KEY` | JSON の `private_key`（`\n` を含む長い文字列。ダブルクォートで囲む） |
| `FIREBASE_STORAGE_BUCKET` | `<demo-project-id>.appspot.com` |

> ⚠️ 「Firebase 初期化エラー」の大半は `FIREBASE_PRIVATE_KEY` の改行崩れ。Vercel ではダブルクォート付きで貼るか、改行をそのまま貼る（本リポは `\n` を改行へ変換する実装）。

### 3-2. LINE（demo アプリ）

| 変数 | 値 |
|------|----|
| `NEXT_PUBLIC_LIFF_ID` | demo の LIFF ID |
| `NEXT_PUBLIC_LIFF_ID_REVIEW` | **demo の LIFF ID（同じ値でOK）** |
| `NEXT_PUBLIC_LIFF_ID_PROD` | **demo の LIFF ID（同じ値でOK）** |
| `LINE_CHANNEL_ACCESS_TOKEN` | demo の Messaging API チャネルのアクセストークン |
| `LINE_CHANNEL_SECRET` | demo の Messaging API チャネルシークレット |

> **コツ**: demo では LIFF ID を3つとも同じ demo 値にする。こうすると `detectEnv()` がURLでどれを選んでも必ず demo チャネルになり、「dev/review/prod のどれが効くか」を気にしなくてよい。

LINE Developers 側の設定:
- **LIFF エンドポイントURL** = demo の Vercel URL（例 `https://<demo>.vercel.app`）。**完全一致**が必要。
- **LINE Login のコールバックURL** も demo URL に。
- 自分を**テスター/開発者**として登録（demo チャネルは審査不要で利用可）。

### 3-3. 自分で生成（demo 専用に新規）
本番と別の値を必ず生成する。
```
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
```
| 変数 | 用途 |
|------|------|
| `SESSION_SECRET` | セッションJWT署名（必須・32文字以上） |
| `CRON_SECRET` | Vercel Cron 認証 |
| `PREVIEW_SECRET` | プレビューモード（任意） |

### 3-4. 管理者ログイン
| 変数 | 値 |
|------|----|
| `NEXT_PUBLIC_GOOGLE_OAUTH_CLIENT_ID` | Google OAuth クライアントID |
| `ADMIN_EMAILS` | 管理者メール（カンマ区切り。自分のメール） |
| `ADMIN_ALLOWED_ORIGINS` | demo の管理URL（CSRF許可）。demo では demo URL を設定 |

### 3-5. URL / ドメイン
| 変数 | demo での値 |
|------|------------|
| `NEXT_PUBLIC_PORTAL_URL` | `https://<demo のURL>` |
| `CUSTOMER_DOMAIN` | **demo では未設定でOK**（未設定＝ドメイン制限なし） |
| `ADMIN_DOMAIN` | 同上（未設定でOK） |

### 3-6. メール・カレンダー（任意・後回し可）
予約や招待メールのテストをする時だけ、**demo 用のテスト値**にする。
| 変数 | 注意 |
|------|------|
| `RESEND_API_KEY` / `RESEND_FROM_ADDRESS` | demo は**自分宛 or テストドメイン**に。本番の顧客に招待メールが飛ばないように |
| `GOOGLE_SERVICE_ACCOUNT_EMAIL` / `GOOGLE_PRIVATE_KEY` | demo 用サービスアカウント |
| `CALENDAR_ID_*`（6施設分） | **demo 用のテストカレンダー**に。本番カレンダーに実イベントが入るのを防止 |

### 3-7. 審査モード（任意）
| 変数 | 値 |
|------|----|
| `ALLOW_REVIEW_MODE` | demo で審査用バイパスを試すなら `true`。**本番では未設定**（OFF） |
| `REVIEW_LINE_USER_ID` / `REVIEW_LINE_DISPLAY_NAME` | 審査ユーザー表示用（任意） |

### 3-8. Square（決済）
決済は無効化中のため**設定不要**（`SQUARE_*` は空でよい）。

## 4. Vercel での設定場所

- **demo が別 Vercel プロジェクトの場合**（推奨・分かりやすい）:
  そのプロジェクトの Settings > Environment Variables に上記をすべて入れる（スコープは Production でOK）。Git 連携は `develop` ブランチを Production として設定すると、`develop` push → demo 自動デプロイになる。
- **demo が本番プロジェクトの Preview の場合**:
  Environment Variables を **Preview スコープ**で設定（同名キーに demo 値）。Production スコープには本番値が入っているので**触らない**。`develop` への push が Preview デプロイになる。

> どちらでも可。**「demo の env に本番値を入れない」**が守れていれば安全。

## 5. 反映 → 検証チェックリスト

1. [ ] demo に環境変数を設定し、**再デプロイ**した（`NEXT_PUBLIC_*` は再ビルド必須）
2. [ ] LINE Developers の LIFF エンドポイントURL = demo URL（完全一致）
3. [ ] demo の LIFF URL を **LINE アプリで開く** → ログインできる
4. [ ] **demo Firebase の Firestore** に自分の `users` / `authorizedUsers` が作られる（＝ demo を指せている証拠）
5. [ ] **本番 Firestore は増えていない**（＝ 分離できている証拠）
6. [ ] `/admin`（demo URL）に `ADMIN_EMAILS` のアカウントでログインできる
7. [ ] 管理画面のユーザー一覧が demo のデータになっている
8. [ ]（任意）招待発行・予約は、Resend / Calendar を demo に向けてから試す

## 6. よくあるつまずき

| 症状 | 原因 / 対処 |
|------|------------|
| 画面が真っ白（LIFF） | LIFF エンドポイントURLが demo デプロイURLと不一致 |
| Firebase 初期化エラー | `FIREBASE_PRIVATE_KEY` の改行崩れ（`\n` / ダブルクォート） |
| env を変えたのに反映されない | 再デプロイしていない（特に `NEXT_PUBLIC_*` は焼き込み） |
| demo なのに本番チャネルに繋がる | 独自ドメインだと `detectEnv()`→prod。LIFF ID を3つとも demo 値にする／`?env=review` を付ける |
| 管理画面に入れない | `ADMIN_EMAILS` 未設定 / `NEXT_PUBLIC_GOOGLE_OAUTH_CLIENT_ID` 不一致 |
| 招待メールが実顧客に届いた | `RESEND_*` が本番値。demo は自分宛/テストに |
| 予約が本番カレンダーに入った | `CALENDAR_ID_*` / `GOOGLE_*` が本番値。demo はテストカレンダーに |

---

## 7. 日常の開発フロー

1. `develop` で作業（または feature ブランチ → `develop`）。
2. push → demo に自動デプロイ → demo で動作確認。
3. 問題なければ `develop` を `main` に fast-forward → 本番反映。
4. 本番の環境変数（Production スコープ）は**本番値のまま据え置き**。demo とは独立。

> 関連: 環境変数の一覧と取得先は `.env.local.example`、機能の現行仕様は `CLAUDE.md` を参照。
