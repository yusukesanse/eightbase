# 環境構築・デプロイ運用ガイド（本番運用しながら開発する）

本番（production）を運用しつつ、安全に修正・新機能開発を行うための環境構成と手順。

> **大原則**: demo 環境の環境変数は **すべて demo のリソース（demo Firebase / demo LINE / demo Vercel）を指す値**にする。
> 本番の値は demo に1つも入れない。これさえ守れば、demo での操作が本番に影響することはない。

---

## 1. 環境マトリクス

| 層 | 用途 | Git ブランチ | Vercel プロジェクト | `NEXT_PUBLIC_APP_ENV` | ドメイン(Cloudflare DNS) | LINE チャネル | Firebase |
|----|------|-------------|----------------------|------------------------|---------------------------|--------------|----------|
| local | 手元開発 | 任意 | なし（`localhost:3000`） | `local`（未設定可） | — | demo | demo |
| **demo** | 結合確認・関係者テスト | `develop` | **`eightbase-demo`（本番と別プロジェクト）** | `demo` | `portal.demo…` / `admin.demo…` → demoのみ | **demo** | **demo** |
| production | 本番 | `main` | **`eightbase-prod`** | `production` | `portal…` / `admin…` → prodのみ | 本番 | 本番 |

- **Vercel は本番と demo を完全に別プロジェクト**にする（同一プロジェクトの Preview スコープは使わない）。各プロジェクトの環境変数は自環境のリソースしか指さない。
- `NEXT_PUBLIC_APP_ENV` が環境の唯一の真実（`src/lib/env.ts`）。demo バナー表示・整合チェック・prebuild検査がこの値で動く。
- `detectEnv()`（`src/lib/liff.ts`）: `localhost`→dev / `*.vercel.app`→review / それ以外（独自ドメイン）→prod。`?env=dev|review|prod` でも明示可。**prod は dev LIFF ID へフォールバックしない**。
- ブランチ運用: `develop` で開発 → demo で確認 → 問題なければ `main` に fast-forward → 本番反映。

---

## 2. いま用意済みのもの（前提）

- Vercel: demo 環境
- Firebase: demo プロジェクト
- LINE: demo ミニアプリ（LIFF + Login、必要なら Messaging API チャネル）

残作業は **環境変数を demo の値で設定し、各サービスの URL を demo に向け、検証する**こと。

## 3. 環境変数の取得元と設定先

`.env.local.example` に全キーと取得先が記載済み。**local は `.env.local`、demo は Vercel demo の Environment Variables** に設定する。
`NEXT_PUBLIC_*` はビルド時に焼き込まれるため、変更後は **再デプロイが必要**。

### 3-1. Firebase（demo プロジェクト）
Firebase Console > プロジェクト設定 > サービスアカウント > 「新しい秘密鍵を生成」で JSON を取得し、以下にマッピング:

| 変数 | 値 |
|------|----|
| `FIREBASE_PROJECT_ID` | JSON の `project_id` |
| `FIREBASE_CLIENT_EMAIL` | JSON の `client_email` |
| `FIREBASE_PRIVATE_KEY` | JSON の `private_key`（`\n` を含む長い文字列。ダブルクォートで囲む） |
| `FIREBASE_STORAGE_BUCKET` | `<demo-project-id>.appspot.com` |

> ⚠️ 「Firebase 初期化エラー」の大半は `FIREBASE_PRIVATE_KEY` の改行崩れ。Vercel ではダブルクォート付きで貼るか、改行をそのまま貼る（本リポは `\n` を改行へ変換する実装）。

### 3-2. LINE（demo アプリ）

| 変数 | 値 |
|------|----|
| `NEXT_PUBLIC_LIFF_ID` | demo の LIFF ID |
| `NEXT_PUBLIC_LIFF_ID_REVIEW` | **demo の LIFF ID（同じ値でOK）** |
| `NEXT_PUBLIC_LIFF_ID_PROD` | **demo の LIFF ID（同じ値でOK）** |
| `LINE_CHANNEL_ACCESS_TOKEN` | demo の Messaging API チャネルのアクセストークン |
| `LINE_CHANNEL_SECRET` | demo の Messaging API チャネルシークレット |

> **コツ**: demo では LIFF ID を3つとも同じ demo 値にする。こうすると `detectEnv()` がURLでどれを選んでも必ず demo チャネルになり、「dev/review/prod のどれが効くか」を気にしなくてよい。

LINE Developers 側の設定:
- **LIFF エンドポイントURL** = demo の Vercel URL（例 `https://<demo>.vercel.app`）。**完全一致**が必要。
- **LINE Login のコールバックURL** も demo URL に。
- 自分を**テスター/開発者**として登録（demo チャネルは審査不要で利用可）。

### 3-3. 自分で生成（demo 専用に新規）
本番と別の値を必ず生成する。
```
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
```
| 変数 | 用途 |
|------|------|
| `SESSION_SECRET` | セッションJWT署名（必須・32文字以上）。**本番/demoで別値** |
| `CRON_SECRET` | Vercel Cron 認証。**本番/demoで別値**（demo cron が本番通知を送らない保険） |
| `PREVIEW_SECRET` | プレビューモード（任意） |

### 3-3b. 環境識別と取り違え防止（必須）
| 変数 | demo での値 | 本番での値 |
|------|------------|-----------|
| `NEXT_PUBLIC_APP_ENV` | `demo` | `production` |
| `EXPECTED_FIREBASE_PROJECT_ID` | demo の Firebase プロジェクトID | 本番の Firebase プロジェクトID |

- `EXPECTED_FIREBASE_PROJECT_ID` は **`FIREBASE_PROJECT_ID` と一致しなければ起動/ビルドが失敗**する（`src/lib/env.ts` / `scripts/check-env.mjs`）。各 Vercel プロジェクトに一度だけ、その環境の正しいIDを設定しておけば、後から誤って別環境の Firebase 認証ブロックを貼っても projectId が食い違って弾かれる。
- `NEXT_PUBLIC_APP_ENV=production` のビルドは、`NEXT_PUBLIC_PORTAL_URL` が `vercel.app`/`localhost` のままだと **prebuild で失敗**する（demo値の混入検出）。

### 3-4. 管理者ログイン
| 変数 | 値 |
|------|----|
| `NEXT_PUBLIC_GOOGLE_OAUTH_CLIENT_ID` | Google OAuth クライアントID |
| `ADMIN_EMAILS` | 管理者メール（カンマ区切り。自分のメール） |
| `ADMIN_ALLOWED_ORIGINS` | demo の管理URL（CSRF許可）。demo では demo URL を設定 |

### 3-5. URL / ドメイン
| 変数 | demo での値 |
|------|------------|
| `NEXT_PUBLIC_PORTAL_URL` | `https://<demo のURL>` |
| `CUSTOMER_DOMAIN` | **demo では未設定でOK**（未設定＝ドメイン制限なし） |
| `ADMIN_DOMAIN` | 同上（未設定でOK） |

### 3-6. メール・カレンダー（任意・後回し可）
予約や招待メールのテストをする時だけ、**demo 用のテスト値**にする。
| 変数 | 注意 |
|------|------|
| `RESEND_API_KEY` / `RESEND_FROM_ADDRESS` | demo は**自分宛 or テストドメイン**に。本番の顧客に招待メールが飛ばないように |
| `GOOGLE_SERVICE_ACCOUNT_EMAIL` / `GOOGLE_PRIVATE_KEY` | demo 用サービスアカウント |
| `CALENDAR_ID_*`（6施設分） | **demo 用のテストカレンダー**に。本番カレンダーに実イベントが入るのを防止 |

### 3-7. 審査モード（任意）
| 変数 | 値 |
|------|----|
| `ALLOW_REVIEW_MODE` | demo で審査用バイパスを試すなら `true`。**本番では未設定**（OFF） |
| `REVIEW_LINE_USER_ID` / `REVIEW_LINE_DISPLAY_NAME` | 審査ユーザー表示用（任意） |

### 3-8. Square（決済）
決済は無効化中のため**設定不要**（`SQUARE_*` は空でよい）。

## 4. Vercel での設定場所

- **demo が別 Vercel プロジェクトの場合**（推奨・分かりやすい）:
  そのプロジェクトの Settings > Environment Variables に上記をすべて入れる（スコープは Production でOK）。Git 連携は `develop` ブランチを Production として設定すると、`develop` push → demo 自動デプロイになる。
- **demo が本番プロジェクトの Preview の場合**:
  Environment Variables を **Preview スコープ**で設定（同名キーに demo 値）。Production スコープには本番値が入っているので**触らない**。`develop` への push が Preview デプロイになる。

> どちらでも可。ただし本タスクの方針は **本番と demo を完全に別 Vercel プロジェクト**にすること。**「demo の env に本番値を入れない」**が守れていれば安全。

## 4b. Cloudflare DNS とドメイン割当（ゴール: demo で admin.demo / portal.demo を見る）

Cloudflare は **DNS/ドメイン管理のみ**に使う（アプリの実体は Vercel）。取り違え防止の鉄則は **1ドメイン = 1 Vercel プロジェクト**。

### 手順
1. **Cloudflare で DNS レコードを作成**（例。実ドメインは適宜読み替え）
   | ホスト | 種別 | 向き先 | プロキシ |
   |--------|------|--------|---------|
   | `portal.example.net`（本番顧客） | CNAME | Vercel 本番プロジェクト | DNS only（まずはグレー雲） |
   | `admin.example.net`（本番管理） | CNAME | Vercel 本番プロジェクト | DNS only |
   | `portal.demo.example.net`（demo顧客） | CNAME | **Vercel demo プロジェクト** | DNS only |
   | `admin.demo.example.net`（demo管理） | CNAME | **Vercel demo プロジェクト** | DNS only |
   - 向き先（CNAME ターゲット）は各 Vercel プロジェクトの **Settings > Domains** が案内する値を使う。
   - まずは **DNS only（グレー雲）** にしておくと Vercel のドメイン検証と LIFF エンドポイント一致がシンプル。WAF/プロキシ（オレンジ雲）は分離が固まってから。
2. **Vercel demo プロジェクトに 2 ドメインを登録**: `portal.demo.example.net` と `admin.demo.example.net` を demo プロジェクトの Domains に追加。**同じドメインを本番プロジェクトに登録しない**。
3. **demo プロジェクトの環境変数でドメイン分離を有効化**（middleware が出し分ける）:
   | 変数 | demo での値 |
   |------|------------|
   | `CUSTOMER_DOMAIN` | `portal.demo.example.net` |
   | `ADMIN_DOMAIN` | `admin.demo.example.net` |
   | `ADMIN_ALLOWED_ORIGINS` | `https://admin.demo.example.net`（CSRF許可） |
   | `NEXT_PUBLIC_PORTAL_URL` | `https://portal.demo.example.net` |
   - `src/middleware.ts` は `ADMIN_DOMAIN` ホストで `/admin/*` のみ許可、`CUSTOMER_DOMAIN` ホストで `/admin/*` をブロックする。これで **portal.demo = 顧客全画面 / admin.demo = 管理全画面** が成立する。
4. **LINE Developers（demo チャネル）**: LIFF エンドポイントURL = `https://portal.demo.example.net`（完全一致）。Login コールバックも同様。

### ゴールの確認
- `https://portal.demo.example.net` を LINE アプリで開く → 顧客側の全画面が demo データで見える。
- `https://admin.demo.example.net/admin` に `ADMIN_EMAILS` のアカウントでログイン → 管理側の全画面が見える。
- どちらの画面でも上部に **「DEMO 環境」バナー** が出る（`NEXT_PUBLIC_APP_ENV=demo` のとき / `src/components/DemoBanner.tsx`）。

> 認証なしで全画面をざっと見たいだけなら、プレビューモード（`/preview`・読み取り専用）も利用可。ただし本ゴールは実ドメインでの動作確認を指す。

## 4c. コード側の多層防御（実装済み）

env の取り違えを「人の注意」ではなく仕組みで止める。

| 防御 | 実装 | 効果 |
|------|------|------|
| 起動時の整合チェック | `src/lib/env.ts` `assertEnvConsistency()` を `firebaseAdmin` 初期化で呼ぶ | `EXPECTED_FIREBASE_PROJECT_ID` 不一致・本番URL不正なら **Firestore 接続前に throw** |
| ビルド前検査 | `scripts/check-env.mjs`（`npm run build` の `prebuild`） | 本番ビルド時に必須env欠落・demo値混入・`ALLOW_REVIEW_MODE=true` を検出して**ビルド失敗** |
| 環境バナー | `src/components/DemoBanner.tsx`（layout に組込済） | demo ビルドにだけ「DEMO 環境」帯。人間の取り違え防止 |
| LIFF 誤用防止 | `src/lib/liff.ts` `detectEnv()` | 本番で dev LIFF ID にフォールバックしない（既存） |

- 手元での確認: `npm run check-env`（`NEXT_PUBLIC_APP_ENV=production node scripts/check-env.mjs` のように環境を変えて試せる）。
- `local`/`demo` ビルドは欠落を **警告のみ**（exit 0）で妨げない。`production` のみハード失敗。

## 6. よくあるつまずき

| 症状 | 原因 / 対処 |
|------|------------|
| 画面が真っ白（LIFF） | LIFF エンドポイントURLが demo デプロイURLと不一致 |
| Firebase 初期化エラー | `FIREBASE_PRIVATE_KEY` の改行崩れ（`\n` / ダブルクォート） |
| env を変えたのに反映されない | 再デプロイしていない（特に `NEXT_PUBLIC_*` は焼き込み） |
| demo なのに本番チャネルに繋がる | 独自ドメインだと `detectEnv()`→prod。LIFF ID を3つとも demo 値にする／`?env=review` を付ける |
| 管理画面に入れない | `ADMIN_EMAILS` 未設定 / `NEXT_PUBLIC_GOOGLE_OAUTH_CLIENT_ID` 不一致 |
| 起動時に「環境変数の取り違えを検出」 | `FIREBASE_PROJECT_ID` ≠ `EXPECTED_FIREBASE_PROJECT_ID`。Firebase設定ブロックを別環境の値で貼っていないか確認 |
| 本番ビルドが prebuild で失敗 | `scripts/check-env.mjs` のメッセージに従う（必須env欠落 / `NEXT_PUBLIC_PORTAL_URL` がvercel.appのまま / `ALLOW_REVIEW_MODE=true` 等） |
| demoバナーが出ない/本番で出た | `NEXT_PUBLIC_APP_ENV` の値ミス（demoは`demo`、本番は`production`）。`NEXT_PUBLIC_*`は再ビルド必須 |
| 招待メールが実顧客に届いた | `RESEND_*` が本番値。demo は自分宛/テストに |
| 予約が本番カレンダーに入った | `CALENDAR_ID_*` / `GOOGLE_*` が本番値。demo はテストカレンダーに |

## 7. 日常の開発フロー

1. `develop` で作業（または feature ブランチ → `develop`）。
2. push → demo に自動デプロイ → demo で動作確認。
3. 問題なければ `develop` を `main` に fast-forward → 本番反映。
4. 本番の環境変数（Production スコープ）は**本番値のまま据え置き**。demo とは独立。

## 8. 3環境モデルへの拡張（確定設計 / これから構築）

> 本章は §1〜§7 を **置き換えるのではなく拡張**する確定設計。用語マッピング: **旧「demo」 = 本章の「staging（検証）」**。
> 新たに **「development（開発）」** 層を足し、**ダミーデータは env フラグで制御**する（ブランチ差分にしない）。

### 8-1. なぜこの形か（根っこの問題）
「ダミーデータをコミットとして持つ」と、次の3つは同時に成立しない:
- (a) ダミーが staging のコードに**コミットとして存在する**
- (b) staging → main を**普通にマージできる**
- (c) main に**ダミーが無い**

→ **(a) を崩す**。ダミーを「コミット差分」ではなく **env フラグで発火する挙動**にすれば、(b)(c) が両立し、`feature → staging → main` が全部ふつうのマージで回る（cherry-pick 不要）。

### 8-2. 環境マトリクス（3層）

| 層 | 用途 | Git ブランチ | Vercel | `NEXT_PUBLIC_APP_ENV` | Firebase | ダミー(`EIGHTBASE_DUMMY_DATA`) |
|----|------|-------------|--------|------------------------|----------|-------------------------------|
| **development** | 手元開発・何してもいいサンドボックス | `feature/*`（ローカル） | なし(`localhost`)／必要なら Preview | `development` | **`eightbase-dev`（専用・新設）** | `on` |
| **staging** | 検証・実機確認・関係者テスト | `staging`（長命） | Staging プロジェクト | `staging` | `eightbase-stg`（旧 demo） | `on` |
| **production** | 本番 | `main` | Production | `production` | `eightbase-prod` | **未設定(=off)** |

- `main` はあなたの図の **master** に相当（リネームせず `main` のまま）。
- 開発(local)は **本番DBを指さない**。staging とも分離するため **dev 専用 Firebase を新設**（dev を荒らしても検証に波及しない）。

### 8-3. ブランチ運用（図のまま全部ふつうのマージ）
```
main ──┬─→ feature/xxx ─(コードレビュー)─→ staging に merge（検証・実機確認）
       │                                      │
       └──────────────── OKなら ──────────────┘→ main に merge（リリース）
```
ダミーはどのブランチにも差分を作らないので `staging → main` も `feature → main` も FF 可能。

### 8-4. ダミーデータ制御（A案 = env ゲート）
1. 判定を `isPreviewMode(req)` から **`isDummyDataEnabled()`（env 連動）** へ付け替える。プレビューモードとダミーを分離する（dev はプレビューではない）。
2. `EIGHTBASE_DUMMY_DATA=on` は **dev / staging の env にだけ**設定。**本番には置かない**。
3. **prebuild 二重ガード**（`scripts/check-env.mjs` に追加）: 本番ビルド（`APP_ENV=production`）で `EIGHTBASE_DUMMY_DATA` が有効なら**ビルド失敗**。
   → 「本番にダミーが出ない」は *push を我慢する運用* ではなく *env で発火しない＋ビルドが弾く* 機械的保証。

### 8-5. 管理者app（admin）の認証
- **production**: 現状どおり **Google OAuth**（`/api/admin/auth` POST = IDトークン検証 ＋ `ADMIN_EMAILS`/Firestore 照合）。
- **staging / development**: Google OAuth は過剰なので **共有パスワード1本**でログイン可にする。
  - Vercel 変数 `ADMIN_SIMPLE_PASSWORD`（非本番 env のみ設定）。一致でセッション Cookie 発行。
  - **本番では無効**（`APP_ENV=production` かつパスワード未設定なら経路ごと閉じる）。prebuild で「本番に `ADMIN_SIMPLE_PASSWORD` がある＝失敗」を足して二重に封じる。
  - ログインページは非本番のときだけパスワード欄を出す（本番は Google ボタンのみ）。

### 8-6. 利用者app（portal）の LINE 認証 — 実機確認の切り分け
LINE/LIFF は HTTPS の固定エンドポイント一致が要るため、**層ごとに役割を分ける**:
- **development（localhost）**: LINE を**バイパスしてモックログイン**（dev 専用・env ゲート）。`liff` を初期化せず固定のテストユーザーでセッション発行。LINEアプリ不要で「何してもいい」サンドボックスにする。
- **staging**: **本番同等の実機確認はここで行う**。**staging 用 LINE チャネル(旧 demo)** ＋ LIFF エンドポイント = staging URL（完全一致）。テスター登録した実機の LINE アプリで開いて確認。
- **production**: 本番 LINE チャネル。

> ねらい: 「実機で LINE 越しに確認」は staging に集約し、development はモックで高速に回す。これで「dev では本番同様の実機確認ができない」問題を、*そもそも dev では実機確認しない（staging でやる）* と整理して解消する。

### 8-7. 構築の順序（少しずつ・問題ごとに）
依存の浅い順。各ステップ単独で完結・検証してから次へ。
1. **ダミー env ゲート**（コード）: `isDummyDataEnabled()` 追加 → 各APIの `isPreviewMode` 由来のダミー分岐を付け替え → prebuild ガード追加。※外部インフラ不要、最初に土台を作る。
2. **admin パスワード認証**（コード＋Vercel変数）: 非本番のパスワード経路を追加。
3. **portal dev モックログイン**（コード）: development のみ LINE バイパス。
4. **dev 専用 Firebase 新設**（インフラ）: `eightbase-dev` 作成 → ローカル `.env.local` を dev 値に。
5. **ブランチ整理**: `develop`→`staging` へ集約、ダミー系コミットは env ゲート版に作り直し。`feature/*` は `main` 起点に。
6. **Vercel/DNS**: staging プロジェクトの env を上表の値に（§3〜§4b 準拠、demo→staging 読み替え）。

> 各ステップは「実装→確認→次」で進める。まとめて一気にやらない。
