# ゲスト利用者へのゲーム機能開放 — 要件定義

> ステータス: **要件定義（実装前）**。本書は合意形成用のドラフト。コード実装は未着手。
> 対象: 麻雀リーグ/CS/ランキング等の「ゲーム機能」を、会員でない**ゲスト利用者**にも開放する。

---

## 1. 背景・目的
- リーグ戦に**予約者（会員）以外も参戦**することになった。
- そのため、**ゲーム機能に限り**ゲスト利用者を開放する。施設予約・掲示板・イベント・ニュース・メンバー一覧は**会員限定のまま**。
- 認可モデルの根幹（「会員かどうか」と「使えるか」が一体化）に手を入れるため、**大幅な改修**になる。

## 2. 用語
| 用語 | 定義 |
|------|------|
| 会員 (member) | メール＋OTPで招待され `authorizedUsers` に登録された従来ユーザー。全機能。 |
| ゲスト (guest) | **ゲスト専用URL**から入った LINE ログインユーザー。ゲーム機能のみ。 |
| OTP招待 | 既存の招待方式。`/api/auth/invite` がパスコードを検証し `authorizedUsers` を作成。 |
| プレビュー/審査モード | 既存の別系統（読み取り専用 / 審査用バイパス）。**今回の対象外・流用しない**。 |

> 「招待がない」= このLINEアカウントが `authorizedUsers` に `active=true` で存在しない＝**OTPで会員登録していない**状態。現状 `liff-login` はこの場合 `needsLinking` で入口を塞ぐ。

---

## 3. 現状（なぜ大改修か）
- **認可**: `requireActiveUser`＝`authorizedUsers.active=true`（招待会員）で一律ゲート。`requireProfileComplete` はさらにプロフィール登録を要求。
- **入口**: `liff-login`（`src/app/api/auth/liff-login/route.ts`）は、`authorizedUsers` に居ない LINE ユーザーへ `needsLinking:true` を返し **session を発行しない**。＝ゲストは入口で弾かれる。
- **クライアント**: `AuthGuard`（`src/components/AuthGuard.tsx`）は未認可→`/login`、`profileComplete=false`→`/setup-profile` へ強制。
- **ナビ**: `RichMenu`（`src/components/RichMenu.tsx`）は5タブ固定・出し分け無し（予約 / Info / メンバー / 掲示板 / マイページ）。
- **既存の `role` フィールドは未使用**（`authorizedUsers` に定義はあるが空）。

→ 「ゲスト」という第3の身分を新設し、**入口フロー・認可・ナビ**を分岐させる。

---

## 4. 確定した要件（意思決定済み）
1. **身分表現**: `authorizedUsers.role`（`"member" | "guest"`）で表現。会員昇格は**同一 `lineUserId` のレコード更新**＝**戦績は継承**される。
2. **入口（ワンタイムURL招待）**: **管理画面からゲスト招待を都度発行** → 対象者のメールへ **ワンタイムURL** を自動送信（既存の会員招待＝メール＋OTP の仕組みを流用）。URLを踏んだ人だけがゲスト登録でき、**1URL=1回**（使用済みは無効）。**登録後はそのLINEアカウントが恒久的にゲストとしてゲーム画面を閲覧可**（URLは初回引き換えのトリガにすぎない）。通常URL/会員フローは従来どおり。
3. **氏名/オンボーディング**: 発行時に管理者が **氏名（displayName）＋メール** を入力 → それが順位表の表示名になる。アイコンは LINE から取得。**ゲスト本人の氏名入力は原則不要**（任意で確認/修正画面を出すかは §13 で要確認）。
4. **戦績の扱い**: ゲストも会員と**同等**に正式順位・M1/M2/M3・CSシードへ算入。

### 暫定デフォルト（推奨・後で上書き可）
- **配信**: Resend でメール送信（既存 `sendPasscodeEmail` を流用/派生）。本文に `https://<portal>/guest?code=<token>` 形式のワンタイムURL。
- **有効期限**: 会員招待と同じ日数（`INVITATION_EXPIRY_DAYS`）を流用。
- **ゲストのマイページ**: **簡易マイページ**（自分のリーグ順位/戦績/申告状況＋会員登録(招待)への導線のみ）。

---

## 5. スコープ（機能別 可否マトリクス）
| 領域 | 会員 | ゲスト | 対応API（現状の認可 → 変更後） |
|------|:---:|:---:|------|
| 麻雀 順位表/ランキング 閲覧 | ○ | ○ | `mahjong/standings`・`games/ranking`（requireActiveUser → **requireGameUser**） |
| リーグ参加表明 | ○ | ○ | `mahjong/entries` GET/POST/DELETE（requireActiveUser / **requireProfileComplete** → **requireGameUser**） |
| 当日の卓 閲覧 | ○ | ○ | `mahjong/tables`（requireActiveUser → **requireGameUser**） |
| スコア申告 | ○ | ○ | `mahjong/tables/[id]/report`（**requireProfileComplete** → **requireGameUser**） |
| 日程 閲覧 | ○ | ○ | `mahjong/schedule`（getSessionUserId のみ・現状でもログインのみ） |
| CS 閲覧/参加 | ○ | ○ | `mahjong/cs`・`games/cs`・`games/[id]/join`（一部 requireActiveUser → **requireGameUser**） |
| 施設予約 | ○ | ✕ | `facilities`・`reservations`（→ **requireMember**） |
| 掲示板 | ○ | ✕ | `posts`（→ **requireMember**） |
| イベント / ニュース | ○ | ✕ | `events`・`news`（→ **requireMember**） |
| メンバー一覧/プロフィール | ○ | ✕ | `members`・`auth/profile`（→ **requireMember**） |
| マイページ | ○ | △ | 会員=全項目 / ゲスト=簡易（リーグ状況のみ） |

> 原則: **ゲーム系は `requireGameUser`、それ以外の会員機能は `requireMember`** に付け替える。`requireMember` は実質これまでの `requireActiveUser` と同じ（＝ guest を除外する点だけが新しい）。

---

## 6. 認可モデル
### 新ヘルパー（`src/lib/auth.ts`）
- `requireGameUser(req)`: session の `lineUserId` を取得し、`authorizedUsers` に `active=true` のレコードがあれば **role に関係なく** 許可（member も guest も可）。**profileComplete は要求しない**。
- `requireMember(req)`: 上記に加え `role !== "guest"`（= member）を要求。会員専用機能で使う。
- （必要に応じて）`requireMemberProfileComplete(req)`: 会員かつ profileComplete（投稿/予約など、現 `requireProfileComplete` 相当を会員に限定）。

### 付け替え方針
- ゲーム系API: `requireActiveUser` / `requireProfileComplete` → `requireGameUser`。
- 会員専用API: `requireActiveUser` → `requireMember`、`requireProfileComplete` → `requireMemberProfileComplete`。
- **既定は安全側**（未知のAPIは `requireMember` のまま＝ゲストに開かない）。ゲストに開くAPIを**ホワイトリスト的に明示**する。

### ゲストの整合性
- スコア自己申告の信頼モデルは会員と同じ（4人合計10万点の整合チェック等は既存ロジックをそのまま使用）。

---

## 7. 入口・セッションフロー（管理者発行 → メール ワンタイムURL）
1. **発行**: 管理画面の「ゲスト発行」で対象者の **氏名＋メール** を入力（既存の招待発行UI `/api/admin/invitations` を `role:"guest"` で拡張）。
2. **レコード作成**: `invitations`（`role/type:"guest"`・トークンハッシュ・`expiresAt`）＋ `authorizedUsers`（`role:"guest"` / `profileComplete` 不問）を**アトミックに作成**（会員招待と同じ batch 機構を流用）。
3. **メール送信**: **Resend** で対象メールへ送信。本文に **ワンタイムURL**（例 `https://<portal>/guest?code=<token>`）。
4. **引き換え**: 受信者がURLをタップ → `/guest` ランディング → LIFF 初期化 → LINE ログイン → 埋め込みトークン＋accessToken を引き換えAPIへ（既存 `/api/auth/invite` のロジックを流用 or `role` 分岐）。
5. **原子的消費**: transaction で `usedAt`/`lineUserId` をセット＝**1回限り**。事前作成済みの guest レコードを当該 `lineUserId` に紐付け、`users` にも氏名/アイコンを作成。
6. **恒久アクセス**: 以降は session＋DB上の `role=guest` が身分。**別端末・再ログインでもURL不要で恒久的にゲーム画面を閲覧可**。使用済みURLの再訪・他者の再クリックは無効（`usedAt`/`lineUserId` 済みで弾く）。

### 重要な実装ポイント（事故防止）
- **トークン消費は「ゲスト紐付け成功時のみ」**（transaction 内で `usedAt`＋`lineUserId` を確定。途中離脱でURLを焼かない）。
- **既存ゲスト/会員が新URLを踏んだ場合**：普通にログインさせ、**新トークンは消費しない**（無駄打ち防止）。
- 会員レコードが既にある lineUserId は会員として扱う（ゲスト発行URLでも会員優先）。

---

## 8. クライアント / ナビゲーション
- **AuthGuard**: ゲスト（role=guest）は `/login` 送りにしない。
  - 氏名未登録 → ゲストオンボーディングへ。
  - ゲーム系ルートのみ許可。会員専用ルート（/reservation, /timeline, /members, /info内の会員機能, /events, /news 等）は**ブロック or 非表示**。
  - `/setup-profile`（会員用3ステップ）はゲストに強制しない。
- **RichMenu**: ゲスト時は**ゲーム中心の簡易ナビ**（例: ランキング / （簡易）マイページ）。会員専用タブは出さない。
- **ルーティング**: ゲストの初期到達先はゲーム（麻雀ランキング）。

---

## 9. データモデル変更
- `authorizedUsers`:
  - `role: "member" | "guest"`（既存レコードは `member` とみなすマイグレーション）。
  - ゲスト: `active:true` / `profileComplete` は未使用（または false 固定）/ `displayName`＝入力氏名 / `pictureUrl`＝LINE / `createdVia:"guest-url"` / `guestOnboarded:boolean`。
- `users`（順位表/卓/参加表明の表示名・アバター取得元）: ゲストにもレコードを作成し `displayName`＝氏名、`pictureUrl`＝LINE。
- `mahjongEntries / mahjongTables / standings`: いずれも `lineUserId` 起点なので、ゲストも**そのまま載る**（戦績同等算入の要件と整合）。

---

## 10. 管理画面
- **ゲスト発行**: 既存の招待発行UI（`/api/admin/invitations`）を拡張し、`role:"guest"` で**都度発行＋メール送信**。氏名＋メールを入力 → ワンタイムURLを自動送信。
- **発行履歴/状態**: 既存の招待一覧（未使用/使用済/期限切れ/取消）に**ゲスト招待も表示**（member/guest を区別）。再送・失効も既存導線を流用。
- **ゲスト可視化**: ユーザー一覧で member / guest を区別表示。
- **会員昇格**: ゲストに対して会員招待（OTP）を発行 → 同一 `lineUserId` のレコードが `role:"member"` 化し、**過去のリーグ戦績を継承**。
- **申告の確認/修正**: 従来どおり（ゲスト含む）。
- （任意）ゲストの除外/無効化（`active:false`）。

---

## 11. セキュリティ・不正対策
- **スパムは構造的にほぼ解消**: 発行者＝管理者が宛先メールを指定し、**1URL=1回**消費。無差別なゲスト増殖は起きない。
- **URL漏えい対策**: メール経由のURLが転送・漏えいすると **first-clicker が消費**してしまうため、**短い有効期限**（`INVITATION_EXPIRY_DAYS` 流用 or ゲストは短め）を推奨。必要なら管理者が失効/再発行。
- **トークンの安全な扱い**: 平文トークンは保存せず**ハッシュのみ保存**（既存 `hashPasscode` 流用）。消費は transaction で原子的に。
- **会員機能の保護**: `requireMember` 既定で、ゲストが会員APIを直叩きしても拒否。
- **env/環境分離**: 既存の多層防御（`assertEnvConsistency` 等）はそのまま。

---

## 12. 非対象（今回やらない）
- ゲストへの予約・掲示板・イベント・ニュース・メンバー一覧の開放。
- 決済（無効のまま）。
- ゲスト専用の別LIFF/別チャネル（暫定デフォルトは同LIFF＋専用パス）。

---

## 13. 未決事項・リスク
- ゲスト本人に**氏名の確認/修正画面**を出すか（管理者入力で足りるか）。
- 引き換えを既存 `/api/auth/invite` に **`role` 分岐で相乗り**するか、`/api/auth/guest-redeem` を**別に立てる**か。
- 会員招待メールと**別テンプレート**（ゲスト向け文面・件名）にするか。
- ゲストURLの**有効期限**（会員招待と同日数か、より短くするか）。
- 既存 `authorizedUsers` への `role` 後付けマイグレーション手順（全件 `member` 化）。
- ゲスト氏名の重複/なりすまし（同名ゲスト）への運用方針。
